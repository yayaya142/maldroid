# ADR-0012: Canonical State Authority and Persistence Invariants (REL-011)

**Date**: 2026-07-14  
**Status**: Accepted  
**Context**: Gate 1 of the reliability program — define the contract before refactoring.

## Decision

### Canonical state

`<case>/.maldroid/state.json` is the single canonical source of truth for all durable investigation
state: findings, notes, TODOs, evidence records, profile, context size, and session metadata.

`notes/FINDINGS.md`, `notes/CASE.md`, `notes/TODO.md`, and `notes/SESSION_SUMMARY.md` are derived
views. They are rebuilt from canonical state and may be re-derived at any time without data loss.
If a view is missing or corrupt, `maldroid case rebuild-views [PATH]` restores it.

No external component — UI, model, or external MCP client — may write directly to these derived
files. All mutations must go through `InvestigationManager`.

### Success and failure semantics

Every state-mutating method on `InvestigationManager` follows this protocol:

1. **Validate** the new state against Pydantic models. Raise `ValidationError` if invalid.
2. **Pre-render** the derived Markdown representation. Raise without touching state if this fails.
3. **Commit to in-memory state** (append to list or update index).
4. **Persist** canonical state JSON and derived view atomically. Roll back in-memory state if
   either write raises.

A successful return from any mutation method means:
- `state.json` reflects the new state.
- All derived Markdown files reflect the new state.
- The in-memory `Case.state` object reflects the new state.

A raised exception means:
- No persistent change was made.
- The in-memory state is identical to its pre-call value.
- The caller may retry the operation safely.

### Revision, idempotency, and locking

State writes use `filelock.FileLock` through `atomic_write_json`. Concurrent writers serialize;
they do not produce partial or interleaved state.

`CaseState` does not currently carry a monotonic revision counter. REL-015 will add optional
client mutation keys to all write tools to prevent duplicate records from model retries.

Until REL-015 is implemented, callers must not assume idempotency on retried tool calls.

### View degradation

If canonical state was committed but view rendering fails (e.g. disk full after writing
`state.json`), `InvestigationManager` raises with the in-memory state already rolled back.

The caller receives a clear exception. The state file is consistent. The derived view may be
stale. Running `maldroid case rebuild-views [PATH]` corrects the view without fabricating content.

### Evidence references

`EvidenceReference.description` is optional. When omitted, a default of
`"Reference to {path}"` is generated by the model validator. This default is safe to store and
renders visibly in Markdown views so the researcher can audit it. Model validation errors include
the field path so the model can repair its call.

### Backup and recovery

No automatic backup of `state.json` is created on every write (this would double write load).
`maldroid case check [PATH]` verifies schema integrity, detects truncated JSON, and offers read-
only recovery mode. A future migration will add schema version guards.

## Consequences

- All mutation methods must pre-render before committing state. This is a minor CPU cost paid once
  per mutation and eliminates the ambiguous "error after commit" failure class.
- Tests can inject a failing renderer and assert that state.json is unchanged.
- Derived views are always consistent with `state.json` after a successful mutation.
- The `_render_findings` static method is retained for backward compatibility but delegates to
  the module-level `_render_findings_document` helper.
